# KB3D E-Commerce Platform — Bounded Contexts, State Machines, and Deterministic GUIDs

This document covers the architectural decisions that define how the KB3D platform structures its three bounded contexts, implements state machines as domain aggregates, and ensures deterministic identity across services.

---

## Bounded Contexts and Polyglot Persistence

The platform separates into three bounded contexts, each with its own persistence strategy chosen for data model fit:

**Catalog** (PostgreSQL) owns physical inventory truth. Two aggregates — Products (SKU-based) and Inventory (PartNumber-based) — have different lifecycles and different stakeholders. Products can exist without inventory (digital goods), and multiple products can reference the same inventory item. PostgreSQL was chosen because the relational model fits the structured product hierarchies and foreign-key relationships.

**Storefront** (MongoDB) owns the customer-facing experience. SellableItems are denormalized read models with polymorphic `SellableItemPayload` — a stocked product has `AvailableStock` and `TargetItemId` fields, a digital product has `DownloadUrl` and `FileFormat`, a service has `Duration` and `DeliveryMethod`. MongoDB was chosen because document-oriented storage suits this variable structure naturally.

**ApiService** is the orchestration layer. It owns no domain data. It exposes HTTP endpoints as thin delegates to domain services, and hosts MassTransit consumers that coordinate cross-domain workflows. This separation means domains never consume each other's events directly — all cross-context choreography flows through ApiService.

---

## State Machines as Domain Aggregates

The core architectural choice is using MassTransit saga state machines as the implementation of DDD aggregates. Rather than traditional entity/repository patterns, each aggregate is a `MassTransitStateMachine<TEntity>` where the entity implements `SagaStateMachineInstance`. State transitions are the only way to modify domain data, and every transition publishes domain events.

**Inventory State Machine** — Four states: Available, OnHold, Backordered, Discontinued. Key behaviors:
- **Idempotent creation**: Duplicate `CreateInventoryRequest` messages return the existing item without error
- **Resurrection**: Creating a Discontinued item transitions it back to Available
- **Backorder auto-recovery**: Increasing quantity on a Backordered item automatically transitions to Available
- **Hold semantics**: Held items block quantity modifications but allow description updates
- **Two-phase deletion**: First delete → Discontinued (soft delete). Second delete on Discontinued → `Finalize()` (hard delete, preserves audit trail)
- **Guarded transitions**: Invalid operations throw domain-specific exceptions with structured metadata (`inventoryId`, `currentState`, `attemptedOperation`)

**Product State Machine** — Three states: Enabled, Disabled, Discontinued. Cross-aggregate request/response for inventory validation during creation — sends `InventoryStatusRequest`, handles Completed/Faulted/TimeoutExpired outcomes. Product creation always succeeds; initial state reflects the inventory validation result. Reacts to inventory events (quantity changed, discontinued, deleted) via correlation by `InventoryId` foreign key.

**SellableItem State Machine** — Four states: Draft, Published, Hidden, Discontinued. Tag management via `ITagRepository`. Polymorphic payload handling with `MaterializePayload()` to convert MassTransit proxy objects to concrete types based on `PayloadType` discriminator.

## Deterministic GUID Generation

Cross-domain entity correlation relies on deterministic GUIDs generated by `DeterministicGuid`. Given the same business key (SKU or PartNumber), the same GUID is produced every time. The implementation is zero-allocation, using `stackalloc` and `Span<byte>` throughout:

```
GUID structure: [Namespace:4][Domain:2][Version:2][XxHash64:8]
```

- **Namespace** (4 bytes): Installation-specific identifier, defaulting to `0x4B625374` ("KbSt" in ASCII)
- **Domain** (2 bytes): Bounded context identifier (1=Product, 2=Inventory, 3=SellableItem)
- **Version** (2 bytes): Schema version for forward compatibility
- **Hash** (8 bytes): XxHash64 of the UTF-8-encoded business key

This enables: same SKU always maps to same ProductId; Product and SellableItem for same SKU have different GUIDs (domain byte differs); test namespaces don't collide with production data; no database round-trips to generate entity IDs.

## Correlation Middleware

For commands that carry a business key (SKU, PartNumber) rather than a CorrelationId, MassTransit's default correlation produces EF Core client-side evaluation warnings. The platform solves this with consume middleware that runs before MassTransit's correlation layer.

`ProductCorrelationMiddleware<TMessage>` checks if the incoming message implements `CorrelatedProductCommand`. If so, and if the message carries a non-empty SKU with an empty CorrelationId, the middleware sets `CorrelationId = DeterministicGuid.FromProductSku(message.Sku)`. This converts the database correlation query from a string comparison to a constant GUID lookup.

The middleware is registered globally via `UseConsumeFilter(typeof(ProductCorrelationMiddleware<>), ctx)` in the RabbitMQ transport configuration, but filters at runtime to only process messages implementing the specific contract interface.

## Exception Handling Across the Message Boundary

When a state machine throws a domain exception, MassTransit serializes it into a `RequestFaultException`. The service layer must reconstruct typed domain exceptions from the fault.

The exception hierarchy uses a base class per aggregate (`InventoryException`, `ProductException`, `SellableItemException`) with concrete subclasses: `NotFoundException`, `ValidationException`, `ConflictException`, and `StateException`. Each exception populates its `Data` dictionary with structured metadata.

Extension methods like `InventoryRequestFaultExceptionExtensions.ToInventoryException()` parse the fault's `ExceptionType` (extracting the class name from the fully-qualified type), then reconstruct the appropriate typed exception from the fault's `Data` dictionary. This means API consumers can write `catch (InventoryStateException ex) when (ex.CurrentState == "Held")` rather than parsing exception message strings.
